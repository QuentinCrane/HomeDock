## 1. 项目定位

**HomeDock** 是一个离线优先、局域网原生的双端系统，围绕一个简单的想法展开：

> 在移动端采集，在局域网内回港，在桌面端整理。

项目由两个核心部分组成：

- **Web Base / 主基地** —— 电脑上的本地基地
- **Android Terminal / 回港终端** —— 手机上的采集与回港终端

这套系统主要面向**私人的本地使用场景**，尤其适合那些没有互联网、不需要互联网，或者有意避免依赖互联网的环境。

## 2. 高层架构

```text
┌─────────────────────────────────────────────────────────────┐
│                        本地局域网                            │
│                                                             │
│   ┌─────────────────┐        ┌──────────────────────────┐   │
│   │ Android         │  HTTP  │ Web Base                 │   │
│   │ Terminal        │ <───→  │ (local service)          │   │
│   │                 │  +     │                          │   │
│   │ • 采集内容      │  SSE   │ • 存储胶囊               │   │
│   │ • 本地草稿      │        │ • 展示与管理             │   │
│   │ • 回港同步      │        │ • 归档与回响             │   │
│   └────────┬────────┘        └────────┬─────────────────┘   │
│            │                           │                     │
│            └──── mDNS / 服务发现 ──────────────────────────┘ │
│                    _returnport._tcp                          │
└─────────────────────────────────────────────────────────────┘
```

### 核心通信层

- 使用 **mDNS / NSD** 进行局域网服务发现
- 使用 **HTTP API** 进行创建、更新和同步
- 使用 **SSE** 向 Web 客户端广播实时事件

## 3. Web Base / 主基地

### 核心职责

Web Base 是整个系统的中心节点，负责：

- 托管 Web UI
- 接收来自 Android 的胶囊上传
- 将元数据存入 SQLite
- 将上传文件保存在本地目录
- 广播内容更新事件
- 提供以下页面：
  - 首页 / 归航大厅
  - 碎片墙
  - 回响
  - 档案馆
  - 待办
  - 设置

### 主要技术栈

- **前端**：React + Vite
- **后端**：Express (Node.js)
- **数据库**：SQLite
- **实时通信**：Server-Sent Events
- **服务发现**：bonjour-service (mDNS)
- **样式系统**：Tailwind CSS + Framer Motion

## 4. Android Terminal / 回港终端

### 核心职责

Android Terminal 被设计为一个轻量的移动采集器，负责：

- 采集文字 / 图片 / 音频胶囊
- 先本地离线保存
- 在局域网中发现 Web Base
- 通过“回港”动作进行同步
- 维护草稿 / 待回港 / 已归档等状态

### 主要技术栈

- **语言**：Kotlin
- **UI**：Jetpack Compose
- **数据库**：Room
- **网络**：Retrofit + OkHttp
- **架构**：MVVM + Repository
- **状态管理**：StateFlow + ViewModel
- **服务发现**：NsdManager

### 主要目录结构

```text
android-terminal/
└── app/src/main/java/com/personalbase/terminal/
    ├── TerminalApp.kt
    ├── MainActivity.kt
    ├── data/
    ├── ui/
    └── nsd/
```

## 5. API 结构

当前后端提供一组小而清晰的 REST 风格接口。

### 胶囊相关接口

| 端点                            | 方法   | 作用                  |
| ------------------------------- | ------ | --------------------- |
| `/api/capsules`               | GET    | 获取胶囊列表          |
| `/api/capsules`               | POST   | 创建胶囊（Multipart） |
| `/api/capsules/:id`           | PUT    | 更新胶囊              |
| `/api/capsules/:id`           | DELETE | 软删除胶囊            |
| `/api/capsules/:id/recapture` | POST   | 克隆胶囊              |
| `/api/capsules/:id/restore`   | POST   | 恢复已删除胶囊        |

### Todo 接口

| 端点                | 方法 | 作用         |
| ------------------- | ---- | ------------ |
| `/api/todos`      | GET  | 获取待办列表 |
| `/api/todos`      | POST | 创建待办     |
| `/api/todos/sync` | POST | 批量同步待办 |

### 系统 / 事件接口

| 端点            | 方法 | 作用           |
| --------------- | ---- | -------------- |
| `/api/events` | GET  | SSE 实时事件流 |
| `/api/status` | GET  | 基地状态摘要   |

## 6. 数据流

### 胶囊创建与回港流程

```text
[用户在 Android 上创建胶囊]
        ↓
[存入 Room]（状态：DRAFT 或本地待同步）
        ↓
[用户点击“回港”]
        ↓
[NsdHelper 发现 _returnport._tcp]
        ↓
[解析服务，得到 host + port]
        ↓
[Retrofit POST /api/capsules]
        ↓
[Express 保存文件与元数据]
        ↓
[Web Base 触发 SSE 广播]
        ↓
[Android 本地状态更新]
```

### 这条链路的意义

它让系统具备三个关键属性：

1. **离线优先** —— 内容先存在，再决定何时同步
2. **本地所有权** —— 内容尽量留在设备和局域网内
3. **低配置负担** —— 服务发现让“回港”更自然

## 7. 服务发现

这套系统的重要目标之一，就是尽量避免手动输入 IP。

### Web 侧

Web Base 通过 mDNS / Bonjour 发布一个本地服务：

- 服务名：`_returnport._tcp`
- 端口：通常为 `3000`

### Android 侧

Android Terminal 使用 `NsdManager` 监听服务：

1. 开始发现
2. 等待找到服务
3. 解析服务
4. 根据 host + port 构造 base URL
5. 用解析后的地址完成同步

这也是为什么“回港”看起来更像一种自然连接，而不是繁琐配置。

## 8. 胶囊状态模型

概念上的胶囊生命周期如下：

```text
DRAFT
  ↓
PENDING
  ↓
ARCHIVED
  ↓
FAVORITED
```

对应的内容流转理解为：

- 先本地创建
- 保持草稿
- 标记为待回港
- 回港成功后归档
- 后续可加入收藏

这个状态模型刻意保持简单，方便理解和维护。

## 9. 数据存储

### Web Base

Web Base 将数据分成两类保存：

1. **SQLite**

   - 胶囊元数据
   - 待办元数据
   - 状态与时间戳
2. **uploads/ 目录**

   - 上传的图片
   - 上传的音频

### Android Terminal

Android 侧使用：

- **Room 数据库** 存结构化元数据
- 本地文件路径保存媒体资源

这形成了一个很实用的组合：

- 元数据进数据库
- 文件留在磁盘
- 通过显式同步完成转移

## 10. 自动初始化

当前架构一个很实用的特点是：启动门槛低。

### Web 侧

首次运行时，系统会自动创建：

- `database.sqlite`
- `uploads/`
- 所需数据表（如果不存在）

### Android 侧

Room 会在初始化时自动创建本地数据库。

这让项目更容易启动，也更适合开源协作。

### UI 方向

理想中的界面气质是：

- 安静

## 12. 实时更新

系统包含一个 SSE 层，用来支持 Web 侧实时更新。

### 事件类型

- `capsule:created`
- `capsule:updated`
- `capsule:deleted`
- `fallback:poll`

### SSE 的作用

SSE 主要用于：

- 将胶囊变化广播给已连接的 Web 客户端
- 保持多页面 / 多视图的一致性
- 减少高频轮询的需要

未来如果 Android 端加入完整事件监听，这条链路还可以继续完善。

## 13. 关键设计模式

### Repository 模式（Android）

Android 侧采用 Repository 结构：

```text
ViewModel → Repository → DAO → Room
```

这样可以把 UI 逻辑和存储 / 网络逻辑分离开。

### 离线优先同步

同步模型遵循：

1. 本地创建 / 更新
2. 立即更新本地 UI
3. 进入待同步队列
4. 基地可用时再同步
5. 服务器确认后标记已同步

### 乐观更新（Web）

Web 端在需要时可以做乐观更新：

1. 先生成临时项目
2. 立即渲染到 UI
3. 发出请求
4. 成功后替换 / 确认
5. 失败时回滚

## 14. 安全说明

这是一个面向私人局域网的系统，所以安全模型是刻意保持简单的。

### 当前考虑

- 可能会为本地 HTTP 有意开启 cleartext
- 默认不暴露到公网
- 默认无账号 / 无认证层
- 文件上传应在服务端做校验
- SQL 查询应使用参数化语句

这种设计适合私有本地环境，但并不是公网部署模型。

---

## 15. 已知限制

当前已知限制包括：

- Android 模拟器的 mDNS 能力有限
- SSE 重连期间可能存在短暂空档
- 目前没有很严格的上传大小约束
- 冲突处理策略仍较简单，偏向最后写入胜出
- UI / 页面结构仍在持续打磨

---

## 16. 为什么选择这条技术路线？

这套架构的选择，主要是为了平衡四件事：

### 1. 本地优先可用性

即使没有互联网，这套系统也应依然有意义。

### 2. 最小基础设施

它应该能直接运行在一台本地机器上，而不是依赖复杂部署。

### 3. 明确的双端分工

手机和电脑应承担不同但互补的角色。

### 4. 便于实验和演化

代码结构应能随着想法变化继续调整。

换句话说：

> HomeDock 不是在试图成为一个云产品。
> 它是在尝试成为一个真正“活着”的、本地的、个人化的小系统。
