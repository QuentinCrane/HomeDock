# 现在情况的说明
一、先说清楚现在这版为什么不对
1. 页面重心是散的

现在首页有：

左边投放台
中间最近碎片
右边待整理
中下两条状态条
最下方大底部导航

看起来很多，但没有一个真正的视觉中心。

所以结果就是：

左右都在抢
中间黑区是空的
页面像“拼出来的功能区”

而不是“我进入了一个基地”。

2. 浅色大卡片把气质做没了

你本来要的是：

私人基地
安静
微科技
低刺激
有一点游戏感

但现在的大白卡 / 浅卡，把整体直接拖向：

轻办公
白板工具
web app dashboard

这和你之前整个方向是反着的。

3. 底部导航太像移动端强行桌面化

桌面端现在底部那条巨大导航栏非常破坏沉浸感。

它会让整页看起来像：

“手机 app 放大到电脑上”

不是“私人基地首页”。

4. 页面缺少“空间组织”

现在所有模块都是：

独立卡片
平铺
同等层级

没有：

前后层级
主次空间
引导动线
中心场域

这就是那块黑显得空的根本原因：
不是没有内容，而是没有结构。

二、这次应该换成什么设计方向

我建议你这次彻底切到这个方向：

“桌面端私人基地工作台”
不是 dashboard，不是卡片拼贴，而是一个“主场域 + 周边舱室”的桌面空间。

核心设计理念：

页面中间必须有一个真正被“使用”的区域
不是所有功能都做成独立白卡
要让内容在空间里“流动”
要有一个“基地中心”
要有一个“工作台”
要有一个“归档区”
要有一个“回响区”
三、首页重做方案
首页不该叫“总览页”

它应该是：

归航大厅 / Home Base
你一进来就：

能投放
能看到最近碎片
能继续整理
能进入回响
能感知基地状态
推荐布局：三段式空间，而不是卡片拼贴
结构总览

桌面端首页改成：

左侧：投放舱

固定宽度，承担输入和快速创建

中央：主工作台

占据最大面积，承担最近碎片、待整理、操作中心

右侧：回响 / 状态 / 归档摘要

作为辅助空间，不抢中心

底部：极细状态轨迹

不是巨大横条，不是底部大导航

更具体一点的布局比例
顶部

一条很薄的顶部状态栏，40px 左右高度即可：

左：项目名 / 当前空间名
右：静默模式、夜间、LAN 状态、设置

不要厚顶栏，不要像办公软件。

左侧：投放舱（320~360px）

这个区域做成一个完整的“输入容器”，而不是白色面板。

内容：

标题：投放台
三个切换入口：
文字
图片
录音
中间是输入区
下方是动作区：
存草稿
投放
最底部是最近一次输入的小提示

视觉建议：

深色主体
内部输入区域稍亮一层
边框克制
重点靠层级，不靠大白块
中央：主工作台（页面视觉核心）

这是当前最缺的东西。

这里不要再只是“最近碎片的卡片列表”。
这里要做成一个真正的 工作面 / 桌面 / 碎片整理台。

建议分两层：

上层：最近碎片墙
占中央区域 60~70%
用 masonry / 不规则网格
展示最近 4~8 个碎片
文本、图片、音频混排
看起来像内容被“摆在台面上”
新内容有轻微高亮
支持点开编辑
下层：待整理区
不是右边那种又高又空的独立卡
而是中央下方的一条“待处理轨道”
显示草稿 / 待整理条目
更像工作流，而不是空栏

这样中间那块黑就不再空了，因为：

它真的变成了主场域。

右侧：回响与档案摘要（280~320px）

右侧不做第三根白色高柱。

改成上下两个“次空间”：

上：回响池预览
一条旧内容
一个“随机唤醒”按钮
更轻、更柔和
可以有微弱暖色
下：档案馆摘要
最近归档时间
归档数
当前筛选入口
一个进入档案馆的动作

右侧角色是：

陪伴主场域，不是抢主舞台。

底部：回港轨迹

改成一条细长的状态轨迹带，高度不要超过 56~64px。

内容：

最近 3~5 条回港事件
自然语言短句
可横向滚动或依次排列

不要做成厚白条。
不要看起来像系统日志。
更像“最近在这里发生了什么”。

# web部分
## web编写指令
现在不要继续微调现有页面了，请直接按下面的方向重做页面设计与结构。

核心问题：
当前页面最大的问题不是元素不够，而是整体仍然是“卡片拼贴式 dashboard 逻辑”，没有形成“私人基地空间”的构图。页面中间大面积黑色空白，说明当前布局没有真正建立主场域。现在的完成度从页面设计和呈现角度远低于目标。

新的设计方向：
把整个 Web 端重做为“桌面端私人基地工作台”，而不是“多个浅色卡片拼在深色背景上”。

必须执行以下调整：

1. 首页重做

* 不要继续保留现在这种左投放、中最近碎片、右待整理、下状态条、底部大导航的结构
* 改成三段式空间：

  * 左侧：投放舱（固定宽度，承担输入和快速创建）
  * 中央：主工作台（占最大面积，展示最近碎片、待整理、编辑入口）
  * 右侧：回响预览 + 档案摘要（辅助空间）
  * 底部：细长回港轨迹条
* 中央区域必须成为绝对主视觉，不允许继续大面积空黑

2. 桌面端去掉底部大导航

* 这个底部 tab bar 太像移动端放大版
* 改成顶部轻导航或页面内入口
* 桌面端不允许再保留这条巨大底部导航

3. 重新定义首页视觉中心

* 当前页面没有真正视觉中心
* 新首页必须让“中央工作台”成为页面重心
* 不是多个等权卡片拼贴
* 不是白色卡片漂浮在黑底上
* 而是“一个可以工作的基地主场域”

4. 重新设计碎片墙

* 不能再是一张孤立小卡 + 大片空白
* 改成真正的内容墙
* 使用 masonry / 不规则多列布局
* 同时展示多条内容
* 支持点击后右侧详情编辑抽屉
* 它必须看起来像“碎片被摆在墙上”，而不是组件 demo

5. 重新设计回响池

* 不能再只有中间一张大卡
* 必须增加空间层次
* 中间主回响内容 + 周围弱化旧片段 + 底部动作条
* 页面应该像“记忆被打捞出来的空间”，不是空白展示页

6. 重新设计档案馆

* 不要表格感
* 改成三段式：

  * 左：筛选
  * 中：条目列表
  * 右：详情与编辑
* 要像私人存放室，不像后台数据库

7. 统一视觉语言

* 保留深色、低刺激、基地感、轻微切角
* 但大幅减少浅色大面板的使用
* 不要让米白大卡统治整个界面
* 内容区域和面板层次要靠深浅层级来建立，不靠大白块
* 背景网格继续保留，但更弱、更安静

8. 页面要优先“可居住感”

* 不是更酷
* 不是更多卡片
* 而是更像一个可以待着、工作、整理内容的地方

9. 动效要求

* 新内容投放后，从左侧投放舱进入中央工作台
* 新碎片进入时有局部高亮
* 编辑抽屉平滑滑出
* 删除时卡片轻缩退
* 回响切换时淡出淡入，不要硬切

10. 先输出再改代码
    先输出：

* 新首页线框
* 新碎片墙线框
* 新回响池线框
* 新档案馆线框
* 顶部导航 / 页面内导航方案
  确认后再真正开始改代码

这次不要再沿着“卡片 dashboard”方向修补。
这次直接朝“桌面端私人基地工作台”重做。

# echart
## 以下是echarts的功能添加，请在完成以上内容以后再继续向下进行
对这个“私人基地 / 回港系统”来说，真正有价值的不是把页面都做成传统统计图，而是把 ECharts 当成“内容组织器 + 氛围器官 + 动态反馈层”。

ECharts 专项规划（并入项目总方案）
一、总体原则

ECharts 在这个项目里不应该承担“后台统计图”角色，而应该承担 3 个角色：

第一，把内容组织得更像一个空间
例如碎片墙、档案层级、回港轨迹，不再是普通列表。

第二，把状态反馈做得更有生命感
例如新碎片回港时的脉冲、高亮、流动线。

第三，把“基地感”做出来
例如基地平面图、房间状态、记忆网络、归档结构。

一句话：

ECharts 不是装饰边角，而是页面里能工作的可视化器官。

二、适合你的 ECharts 使用边界
适合用 ECharts 的地方
首页的基地核心状态层
首页的回港轨迹条
碎片墙的组织视图
回响池的“记忆流”
档案馆的层级归档
基地房间 / 空间结构图
日历式历史活跃分布
不适合用 ECharts 的地方
主输入框
长文本编辑器
设置页表单
详情编辑抽屉主体
普通 CRUD 列表本体

所以正确方式不是“整个页面都用 ECharts 画”，而是：

页面骨架仍然用 React UI，关键的空间结构和动态反馈用 ECharts。

三、最值得纳入项目的 ECharts 模块

下面这些，是我建议你正式纳入设计方案的。

1. 首页：基地核心脉冲层
目标

让首页不再只是几张卡片，而是有一个“基地正在呼吸”的中心。

建议用法

用 Gauge + EffectScatter + Custom 组合。

作用
Gauge 只取“环形核心”感，不做传统大表盘
EffectScatter 做“新胶囊回港时的轻微脉冲”
Custom 画一些克制的核心边缘层或分段刻度
页面落点

放在首页中央主工作台的上层或背景层，作为“基地核心反应器”。

交互
默认慢速呼吸
Android 回港成功后，核心脉冲一次
如果 3 分钟内有新内容，核心边缘轻微高亮
静默模式下，脉冲变慢、亮度降低
注意

不要做成传统 dashboard 仪表盘，不显示夸张百分比，不写 STATUS/HLTH 之类的字。

2. 首页：回港轨迹条
目标

把现在底部那条“普通事件列表”升级成真正有动线感的“回港轨迹”。

建议用法

用 Lines + Scatter 或极简 Custom Series。

作用
每个事件是一个节点
不同类型内容用不同点样式
节点间用低对比线串起来
hover 显示：某条文字碎片、某张图片、某段音频是什么时候回来的
页面落点

首页底部，取代现在纯文字横条。

交互
最近事件默认显示 5~8 个
hover 某节点，显示摘要
点击进入对应碎片详情
新回港事件从右侧滑入并接到轨迹尾部
视觉要求

不是技术 syslog，而是“最近发生的事的轨迹带”。

3. 首页 / 碎片墙：碎片组织视图
目标

让碎片墙不再只是普通卡片网格，而是真正有“内容结构感”。

建议用法

主方案：Treemap
增强方案：Custom Series 做混合碎片卡

作用

Treemap 非常适合做：

不同大小的碎片块
按类型 / 时间 / 状态切分
更像“被切分的墙面”而不是商品列表
页面落点

有两种落法：

方案 A：碎片墙主视图就是 Treemap

适合更实验、更视觉化的版本。

方案 B：常规碎片卡网格 + 一个“切换为结构视图”按钮

这个更稳妥，也更用户友好。

交互
点击块，打开右侧编辑抽屉
hover 显示摘要
新碎片块有轻微亮边
可按草稿 / 待整理 / 已归档重新布局
为什么适合你

Treemap 天然不像后台表格，反而很像“私人资料在墙上占据空间”。

4. 回响池：记忆流
目标

让回响池不再是一张孤零零的大卡，而真的有“回忆浮现”的感觉。

建议用法

ThemeRiver 作为核心背景层。

作用

ThemeRiver 很适合表达：

过去一段时间里不同类型内容的浮沉
回响内容在时间中的流动
记忆不是点，而是一条缓慢流过的东西
页面落点

回响池中央主卡的后方或下方，作为“记忆流底层”。

交互
当前被唤醒内容对应 ThemeRiver 的一个时间段高亮
点击河流某段，可以抽取那段时间附近的内容
静默模式下河流更弱、更淡
优点

比雷达、扫描线、控制台噪点更适合“情绪空间”。

5. 回响池：记忆关系网

让“当前这条回响”和其他旧内容形成关系，不再只是单点显示。

作用

## echart真正对应的操作
在之前的页面重构方案基础上，再正式加入 ECharts 设计层。ECharts 不是装饰边角，而是页面里的“空间组织器 + 动态反馈层”。

请把 ECharts 用在以下地方：

1. 首页中央加入 BasePulseChart

* 用 ECharts 做一个克制的基地核心脉冲层
* 不要传统 dashboard 仪表盘
* 可用 Gauge + EffectScatter + 少量 Custom
* 新胶囊回港时要有一次轻微脉冲

2. 首页底部回港轨迹改成 ReturnTraceChart

* 用 Lines + Scatter 或 Custom 做“回港轨迹条”
* 不要普通文字列表
* 新事件进入时有动态更新
* 点击节点可打开对应碎片详情

3. 碎片墙加入 FragmentTreemapView

* 作为碎片墙的结构视图模式
* 使用 Treemap 展示文字/图片/音频碎片的空间占用与层级
* 点击块可进入右侧编辑抽屉

4. 回响池加入 EchoRiverChart

* 用 ThemeRiver 做记忆流背景层
* 当前被唤醒内容对应一段高亮
* 风格必须柔和，不能控制台化

5. 档案馆加入两个可视化总览

* ArchiveCalendarHeatmap：显示每天回港/记录热度
* ArchiveSunburstOverview：显示按时间/类型/状态的归档层级

6. 不要把 ECharts 做成传统统计图

* 隐藏不必要的坐标轴
* 减少 tooltip 干扰
* 保持深色、低刺激、空间感

* 不要一开始就做满，先从首页核心、回港轨迹、碎片结构视图开始

# 安卓app
## app大致设计

Web 端是“基地本体”，负责安放、整理、回看。
安卓端不应该做成它的缩小版，而应该是：

> **回港终端 / 采集器 / 随身入口**

也就是说，安卓端的核心不是“浏览整个基地”，而是：

* 快速记录
* 轻量整理
* 明确同步
* 随时知道自己和基地是否连上
* 回宿舍后一键回港

下面我给你一份**安卓 App 专项详细设计方案**，可以直接并入总规划。

---

# 一、安卓端产品定位

## 1.1 一句话定义

安卓端不是“第二个 Web”。

它应该是：

> **一个低门槛、强反馈、可离线、偏采集与回港的私人终端**

---

## 1.2 安卓端只做 4 件核心事

第一，**采集**
在外面快速写一句、拍一张、录一段。

第二，**暂存**
离线可用，本地先存，不强迫立刻上传。

第三，**回港**
回到局域网后，把本地胶囊送回基地。

第四，**轻整理**
支持改标题、改备注、删、恢复、改状态，但不承担重度档案管理。

---

## 1.3 安卓端不该是什么

不要做成：

* Web 端的缩小版
* 复杂资料库
* 重内容消费端
* 仪表盘
* 复杂统计中心

安卓端应该偏：

* 快
* 直觉
* 单手友好
* 离线优先
* 强反馈
* 仪式感强

---

# 二、安卓端的整体设计方向

## 2.1 气质

安卓端比 Web 端更“贴身”，所以气质要稍微更亲近一些。

关键词：

* 安静
* 克制
* 便携
* 仪式感
* 轻微中二
* 回港感
* 终端感，但不是黑客终端

---

## 2.2 核心体验原则

### 原则 1：一步到位

用户打开 app 后，最好 **1 步开始输入，2 步完成保存**。

### 原则 2：同步必须明确

用户必须清楚知道：

* 有没有连上基地
* 现在能不能回港
* 回港成功了没
* 哪些没成功

### 原则 3：移动端比桌面端更“轻”

不要塞太多二级信息。
安卓端只展示当前最重要的信息。

### 原则 4：状态比页面更重要

安卓端更像“状态机”：

* 未连接
* 已发现基地
* 待回港
* 同步中
* 已送达
* 失败待重试

页面只是这些状态的载体。

---

# 三、信息架构

我建议安卓端控制在 **4 个一级入口**，不要更多。

## 3.1 一级导航

### 1. 投放

主页面。
负责快速输入、上传、录音、保存。

### 2. 待回港

显示本地待同步内容，负责回港动作。

### 3. 胶囊

浏览本地胶囊，做轻编辑、改状态、删、恢复。

### 4. 设置

网络、静默、同步、动效、通知等。

---

## 3.2 不建议单独做“回响池”主 Tab

回响更适合 Web 端。
安卓端可以做一个轻量入口，比如：

* 在胶囊页加“随机回看”
* 在首页加一个“今日回响”

但不建议做成安卓主页面中心。

# 四、安卓端页面设计方案

---

## 1. 投放页 / Capture

这是安卓端的首页，也应该是最强页面。

### 页面目标

用户一打开 app，就可以立刻：

* 输入一句话
* 拍照/选图
* 录音
* 保存草稿
* 标记为待回港

---

## 1.1 结构建议

```text
顶部栏
[私人基地终端]                     [连接状态点]

中部主区
[模式切换]
文字 | 图片 | 录音

[内容输入区 / 图片预览区 / 录音区]

下部动作区
[存草稿]        [投入待回港]

底部状态提示
“本地已存 3 枚胶囊，待回港 2 枚”
```

---

## 1.2 三种模式

### 文字模式

* 大输入框
* 支持多行
* 自动保存草稿
* 可加一句备注

### 图片模式

* 拍照 / 从相册选
* 图片预览
* 可加一句描述
* 支持删除重选

### 录音模式

* 按住录 / 点击录
* 显示波形或时长
* 停止后可试听
* 可加一句备注

---

## 1.3 设计重点

### 输入区必须大

不要把输入框做成小组件。
它应该是首页的主角。

### 按钮要层级清晰

* 主按钮：投入待回港
* 次按钮：存草稿

### 成功反馈要明确

保存后不能只是按钮变一下。
建议：

* 卡片轻微收拢
* 底部出现“已收入本地舱”
* 待回港数 +1

---

## 2. 待回港页 / Dock Queue

这是安卓端最有“仪式感”的页面。

### 页面目标

用户在这里看到：

* 有哪些内容待回港
* 当前是否已发现基地
* 一键回港
* 进度和结果

---

## 2.1 页面结构

```text
顶部栏
[待回港]                  [基地连接状态]

中部
[待回港胶囊列表]
胶囊1
胶囊2
胶囊3

底部固定区
[当前状态]
已发现基地 / 未连接 / 同步中 / 已完成

[回港主按钮]
RETURN TO DOCK / 发起回港
```

---

## 2.2 视觉重点

### 回港按钮必须是主视觉

不能藏在列表里。
这是整个页面的仪式核心。

### 状态要非常明确

推荐状态文案：

* 未发现基地
* 已发现基地
* 正在对接
* 正在回港
* 基地已接收
* 本次有 1 项失败

---

## 2.3 回港动作设计

点击“回港”后：

### 阶段 1：锁定

* 主按钮变为不可再次点击
* 页面轻微收紧
* 顶部状态点开始脉冲

### 阶段 2：发送

* 列表里的胶囊按顺序进入“发送中”状态
* 可以显示一个纵向流动动画或圆形进度环

### 阶段 3：完成

* 成功的胶囊变成“已送达”
* 失败的保留并标红弱提示
* 页面底部显示：

  * “本次回港完成，已送达 3 枚”
  * “1 枚待重试”

---

## 3. 胶囊页 / Capsules

这个页面承担“轻整理”。

### 页面目标

用户可以：

* 看本地胶囊
* 筛选
* 改标题
* 改备注
* 改状态
* 删 / 恢复
* 收藏

---

## 3.1 页面结构

```text
顶部栏
[胶囊]          [筛选] [搜索]

主列表
胶囊卡片 1
胶囊卡片 2
胶囊卡片 3

点击任意卡片 -> 底部抽屉展开详情
```

---

## 3.2 列表设计

卡片不要做得像后台列表。
更像“本地收纳盒”：

每个卡片显示：

* 类型图标
* 内容摘要
* 时间
* 当前状态
* 是否已同步

支持：

* 左滑删除
* 右滑收藏/归档
* 点击展开详情

---

## 3.3 详情抽屉

内容：

* 原内容
* 备注
* 时间
* 状态切换
* 删除 / 恢复
* 收藏 / 取消收藏
* 重新投入待回港

安卓端详情不要太重，保持“轻编辑”。

---

## 4. 设置页 / Settings

这个页面虽然次要，但要设计得完整。

建议包含：

### 同步设置

* 自动发现基地
* 连接状态
* 上次回港时间
* 重试策略

### 体验设置

* 静默模式
* 动效强度
* 轻震动
* 提示音

### 内容设置

* 自动保存草稿
* 草稿保留时长
* 图片压缩
* 录音质量

# 五、安卓端交互设计

### 柔和

不是“工具 app 的生硬交互”，而是有一点终端感。

主要操作区应落在中下半屏。

胶囊列表支持：

长按胶囊卡片：

# 六、安卓端动效设计

你前面已经强调想要丰富的动效，那安卓端其实特别适合做“有仪式感但不重”的动效。

---

## 1. 投放动效

### 输入开始

* 输入区轻微发亮
* 当前模式 tab 高亮

### 保存草稿

* 按钮缩放
* 输入内容缩成一张小胶囊卡
* 底部出现“已收入草稿舱”

### 投入待回港

* 卡片向下或向右滑入“待回港列表”
* 待回港角标 +1

---

## 2. 回港动效

这是安卓端最该花心思的地方。

### 主按钮

* 普通状态：低频呼吸
* 已发现基地：边缘亮起
* 同步中：转为环形进度态
* 完成：中心点收缩一下再恢复

### 胶囊列表

同步时：

* 每张卡片依次被“吸走”
* 状态从灰 → 蓝 → 绿
* 失败项停留并抖一下

### 页面背景

回港过程中可以加一层极弱的流动光，不要太科幻。

---

## 3. 状态指示动效

顶部连接状态点建议做成：

* 未连接：静止灰点
* 已发现：慢脉冲青点
* 同步中：快速脉冲
* 成功：稳定亮点
* 失败：红点轻闪

---

## 4. 列表动效

* 新胶囊插入：自下而上淡入
* 删除：向侧边缩退
* 恢复：从底部回弹出现
* 展开详情：底部抽屉平滑上推

# 七、反馈设计

安卓端反馈必须比 Web 更强，因为它是“操作终端”。

---

## 1. 视觉反馈

### 成功

* 小型绿色/青绿色状态条
* 卡片边缘变亮
* 待回港数更新

### 失败

* 明确失败项标识
* 重试入口显眼
* 不要只是 toast 一闪而过

### 进行中

* 同步中进度环
* 列表内单项进度
* 页面顶部状态点变化

---

## 2. 触觉反馈

安卓端建议加入轻震动：

* 保存草稿：轻震一下
* 发起回港：更明显一点
* 同步完成：短双震
* 同步失败：短促单震

---

## 3. 声音反馈

建议做成可关闭选项：

* 回港成功的轻提示音
* 回港失败的弱警示音

默认不要太响，也不要太像系统通知。

# 八、安卓端视觉设计

---

## 1. 和 Web 端的关系

安卓端不应该完全复制 Web 端。

关系应该是：

* Web：基地本体，更完整、更空间化
* Android：终端，更紧凑、更直接、更可操作

所以安卓端视觉应该：

* 更简洁
* 更聚焦
* 更适合触摸
* 保留基地感，但不要做得太“桌面化”

---

## 2. 视觉层级

### 背景

* 深灰蓝
* 不要纯黑
* 网格极淡甚至可无

### 面板

* 使用中等亮度深灰面板
* 层级清楚
* 不要大面积浅卡

### 强调色

* 青蓝作为主高亮
* 绿色表示回港成功
* 暗红表示错误

---

## 3. 组件风格

### 卡片

* 轻微圆角 + 少量切角
* 不要太像 iOS 风格白卡
* 更像“终端模块”

### 按钮

* 主按钮面积大
* 文字清楚
* 主次分明

### 图标

* 简洁线性图标
* 不要太拟物
* 类型图标要一眼能认出

# 九、安卓端加入图表/可视化的建议

你前面提到 ECharts，这个主要还是更适合 Web。
**安卓端不建议直接强塞 ECharts WebView**，会破坏原生体验。

安卓端如果想加“可视化”，建议做成 **原生轻量可视反馈**：

### 可做的原生可视化

* 回港进度环
* 小型时间轨迹条
* 胶囊类型分布环
* 周活跃度迷你条带
* 基地连接雷达点（轻量版）

这些可以用：

* Compose Canvas
* 自定义绘制
* 轻量图形组件
  来做。

结论是：

> **Web 用 ECharts 做空间级可视化**
> **Android 用原生 Compose 做终端级可视反馈**

这样最合理。

* 深灰蓝黑
* 比 Web 更纯净一点
* 少纹理或极弱纹理

### 面板

* 中深灰蓝
* 比背景亮一层
* 不要大面积纯白卡

### 主文字

* 柔和冷白

### 次文字

* 灰蓝

### 主强调

* 低饱和青蓝

### 成功色

* 柔和青绿

### 警告/失败

* 低饱和暗红 / 暗橙

---

## 4.3 字体与图标

### 字体

* 主体内容用高可读常规字体
* 辅助状态、时间、编号可少量等宽字体
* 不要全 App 都是终端字体

### 图标

* 线性、清晰
* 不要太拟物
* 建议统一用一套细线风格

类型图标建议：

* 文本：纸页/横线图标
* 图片：相片框
* 音频：波形/麦克风
* 待回港：上行箭头或舱门
* 已归档：盒子/抽屉
* 收藏：星标
* 失败：断链/警示三角

# 六、页面详细线框与设计

# 1. 采集页 / Capture

这是 App 首页，也是最重要的页面。

## 页面目标

用户打开后：

* 不思考
* 直接输入
* 2 步完成保存

---

## 1.1 页面线框

```text
┌────────────────────────────────┐
│ 回港终端                ●已发现基地 │
├────────────────────────────────┤
│ [文字] [图片] [录音]             │
│                                │
│ ┌────────────────────────────┐ │
│ │ 输入区 / 图片预览 / 录音区    │ │
│ │                            │ │
│ │ 文字模式：多行输入框          │ │
│ │ 图片模式：预览缩略图 + 描述    │ │
│ │ 录音模式：波形 + 时长 + 备注   │ │
│ └────────────────────────────┘ │
│                                │
│ 最近一次输入：刚刚保存为草稿      │
│                                │
│ [存草稿]             [投入待回港] │
├────────────────────────────────┤
│ 本地草稿 2   待回港 3   已归档 12  │
└────────────────────────────────┘
```

---

## 1.2 文字模式

### 功能

* 多行输入
* 自动暂存
* 可加一句备注
* 可存草稿
* 可直接投入待回港

### 体验重点

* 输入框必须大
* 一打开就能打字
* 输入不需要经过二级页面

### 细节

* 如果 5 秒无操作，自动草稿保存
* 用户继续输入时，不要频繁 toast
* 保存草稿后只给轻反馈

---

## 1.3 图片模式

### 功能

* 拍照
* 从相册选择
* 预览
* 添加一句描述
* 删除重选
* 存草稿 / 待回港

### 体验重点

* 预览图要明显
* 拍照后不应立刻上传
* 用户能看见“这张图会和一句说明一起保存”

### 细节

* 大图自动压缩
* 预览支持裁切缩略
* 图片来源清楚标识“来自相机 / 相册”

---

## 1.4 录音模式

### 功能

* 点击开始/停止
* 或按住录（任选其一，建议点击录更稳）
* 显示时长
* 可试听
* 可加备注
* 存草稿 / 待回港

### 体验重点

* 波形不是必须复杂，但至少要有“录到东西了”的感觉
* 停止后立即可回放
* 不要录完之后跳到别页

### 细节

* 录音失败时给明确提示
* 如果权限未开，给一次性说明，不要报系统感错误

---

## 1.5 页面主按钮设计

### 主按钮：投入待回港

* 这是主动作
* 文案不要叫“上传”
* 建议：`投入待回港` / `收入待回港舱`

### 次按钮：存草稿

* 明确是本地行为
* 建议：`存草稿`

### 状态变化

当内容不完整时：

* 主按钮 disabled
* 但视觉仍清楚可见

# 2. 待回港页 / Dock Queue

这是安卓端最有“回港仪式感”的页面。

## 页面目标

让用户明确知道：

* 现在能不能回港
* 有多少东西待回港
* 回港成功了多少
* 哪些失败了

---

## 2.1 页面线框

```text
┌────────────────────────────────┐
│ 待回港                    ●已发现基地 │
├────────────────────────────────┤
│ 基地状态：已发现，可发起回港       │
│ 待回港：3 枚                     │
│                                │
│ ┌────────────────────────────┐ │
│ │ 胶囊 1                      │ │
│ │ 文字摘要 / 时间 / 状态        │ │
│ └────────────────────────────┘ │
│ ┌────────────────────────────┐ │
│ │ 胶囊 2                      │ │
│ │ 图片摘要 / 时间 / 状态        │ │
│ └────────────────────────────┘ │
│ ┌────────────────────────────┐ │
│ │ 胶囊 3                      │ │
│ │ 音频摘要 / 时间 / 状态        │ │
│ └────────────────────────────┘ │
│                                │
│ [ 发起回港 ]                    │
│                                │
│ 最近结果：上次成功送达 2 枚       │
└────────────────────────────────┘
```

---

## 2.2 状态设计

这个页面必须围绕状态变化做设计。

### 状态 1：未发现基地

显示：

* 灰色状态点
* 文案：未发现基地
* 按钮不可用，但保留位置
* 提示：连接到基地网络后可发起回港

### 状态 2：已发现基地

显示：

* 青色慢脉冲点
* 文案：已发现基地
* 主按钮激活

### 状态 3：回港中

显示：

* 状态点快速脉冲
* 主按钮变进度态
* 每条胶囊显示“发送中”

### 状态 4：已送达

显示：

* 成功胶囊打勾 / 柔和青绿
* 文案：基地已接收

### 状态 5：部分失败

显示：

* 失败胶囊高亮但不要太刺眼
* 保留一键重试入口

---

## 2.3 回港主按钮动效

这个按钮是全页主角。

### 默认态

* 微弱呼吸
* 不张扬

### 已发现基地

* 边缘轻亮
* 提示“可发起回港”

### 点击后

* 按钮收缩
* 转成圆弧进度或填充态
* 文案变为：`正在回港`

### 完成后

* 按钮短暂变成完成态
* 然后恢复为普通态

---

## 2.4 单个胶囊同步反馈

每个待回港卡片要有自己的微状态：

* 待发送
* 正在发送
* 已送达
* 失败待重试

如果列表很多，用户也能看清到底哪条出了问题。

# 七、安卓端状态机设计

这部分非常重要，直接决定 App 是否“顺手”。

---

## 1. App 全局状态机

```text
未启动
  ↓
初始化
  ↓
读取本地数据
  ↓
发现网络状态
  ├─ 未发现基地
  └─ 已发现基地
        ↓
     可回港
```

---

## 2. 胶囊生命周期状态机

```text
新建中
  ↓
草稿
  ├─ 继续编辑
  ├─ 删除
  └─ 投入待回港
         ↓
      待回港
         ├─ 回港中
         │    ├─ 已送达
         │    └─ 失败待重试
         ├─ 改回草稿
         ├─ 删除
         └─ 收藏
```

建议字段至少有：

* `status`
* `syncState`
* `isFavorite`
* `lastAttemptAt`
* `lastSyncedAt`
* `errorMessage`

---

## 3. 基地连接状态机

```text
未扫描
  ↓
扫描中
  ├─ 未发现
  ├─ 发现基地
  │    ├─ 可连接
  │    └─ 不可连接
  └─ 错误
```

UI 上要一眼能看出来当前在哪个状态。

---

## 4. 回港任务状态机

```text
空闲
  ↓
准备发送
  ↓
逐条发送
  ├─ 成功
  ├─ 部分失败
  └─ 全部失败
  ↓
等待基地确认
  ↓
完成
```

如果你们后续加 Android SSE 客户端，这条链就更完整了，因为 Android 可以真的知道“基地已收到”，这也是你们待规划里的高优先级方向之一。

---

## 5. 录音子状态机

```text
空闲
  ↓
请求权限
  ├─ 拒绝
  └─ 允许
       ↓
     录音中
       ├─ 暂停（可选）
       ├─ 停止
       └─ 取消
             ↓
         试听/保存
```

# 九、动效设计细则

你们现有安卓端已经做过按钮涟漪、列表交错入场、滑动删除、同步进度圆弧、状态脉冲、底部弹窗等，这很好。接下来不是“更多动画”，而是“更统一的动画语言”。

---

## 1. 采集页动画

### 模式切换

* 文字/图片/录音切换时，面板横向滑动或淡切
* 不要整屏切页

### 保存草稿

* 当前输入内容轻微缩成一枚“胶囊”
* 胶囊滑入底部状态区
* 状态计数 +1

### 投入待回港

* 胶囊从输入区滑向“待回港”提示
* 底部显示轻提示：已收入待回港舱

---

## 2. 待回港页动画

### 发起回港

* 主按钮从长条态收缩为进度态
* 状态点由慢脉冲变快脉冲
* 列表中胶囊卡逐张进入发送态

### 单个胶囊发送成功

* 卡片边框高亮一下
* 图标变成成功态
* 可轻微上移后归位

### 失败

* 卡片抖一下
* 保持原位
* 出现“重试”小动作

---

## 3. 胶囊页动画

### 新卡片进入

* 从底部淡入并轻微上滑

### 点开详情

* 底部抽屉平滑上推
* 背景轻暗

### 删除

* 卡片向左滑退 + 透明度降低

### 恢复

* 卡片从下方重新滑入

---

# 十、反馈设计细则

安卓端反馈要比 Web 更强，但不能烦。

---

## 1. 即时反馈

* 按钮按下缩放
* 输入有效高亮
* 图片上传成功的预览出现
* 录音完成的时长显示

## 2. 过程反馈

* 正在扫描基地
* 已发现基地
* 正在回港
* 正在等待基地确认

## 3. 结果反馈

* 已保存草稿
* 已投入待回港
* 本次回港已完成
* 基地已接收
* 1 枚失败待重试

---

## 4. 触觉反馈建议

* 存草稿：轻震
* 投入待回港：轻震
* 发起回港：中等震动
* 同步完成：双短震
* 失败：短促单震

# 十一、和 Web 的角色边界

这个很重要，避免安卓端做得太重。

## 安卓端做什么

* 采集
* 暂存
* 回港
* 轻整理
* 查看本地状态

## Web 端做什么

* 深度整理
* 大量浏览
* 结构视图
* ECharts
* 档案总览
* 回响空间

所以安卓端不要追求“所有东西都能在手机上做完”。
它应该更像：

> **把外面的自己带回来的一只手持终端**

---

# 十二、开发落地优先级

## Phase 0

* 把连接状态和回港状态机做清楚
* 把页面骨架立住

## Phase 1

* 采集页完整实现
* 待回港页完整实现
* 详情抽屉打通

## Phase 2

* 胶囊页轻整理
* 设置页完善
* 同步反馈打通

## Phase 3

* Android SSE 客户端
* 同步历史
* 上传进度
* 冲突处理

这也和你们当前文档里的优先级基本一致：SSE 客户端、同步状态实时反馈、离线队列管理、媒体上传进度这些都已经被列成安卓端待增强项。

---

# 十三、最终完成标准

安卓端完成后，应该达到：

## 外观上

* 深色、安静、统一
* 像终端，但不是黑客工具
* 比 Web 更紧凑、更贴身

## 体验上

* 打开就能采集
* 连接和同步状态一眼可知
* 回港动作有明确仪式感
* 不会让人迷路

## 功能上

* 文字/图片/音频都能顺畅采集
* 草稿 / 待回港 / 已送达 流转清楚
* 轻编辑可用
* 失败可重试
* 状态反馈完整

## 同步上

* 用户能明确知道内容已被基地收到
* 后续若接入 Android SSE，也能把“基地确认”做得更闭环
